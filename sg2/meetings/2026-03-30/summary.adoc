==== 2.6.1 Discussion: ICAO Aviation Formats

Peter Thorne raised a concern about the note that IWXXM/ICAO formats are out of scope, arguing that high-frequency aviation weather observations (e.g., thunderstorm downdraft data) are scientifically valuable and should not be implicitly excluded from national archives. David confirmed the intent is not to exclude them from archiving, and that the note is only in the comment replying to Xavier rather than the functional requirements. David noted that from a political perspective, keeping the document simple and not explicitly mentioning IWXXM and other aviation formats would avoid opening discussion over the scope of WMO versus ICAO. Peter was satisfied provided there is no explicit exclusion. William asked whether there were similar political complications for IOC and ocean data; David replied that there were not — marine data collected in the context of WMO, IOC, or GOOS programmes is in scope without complication.

==== 2.6.2 Discussion: Homogenization in Figure 1 and Introductory Text

Denis raised a concern that "homogenization" appears prominently in Figure 1 and in the introductory text of Chapter 1, giving the impression it is a core or mandatory function of a CDMS, when in fact it is a recommended function and not widely implemented in current CDMS products. He noted that normals data, for example, might be a more fundamental function to highlight.

David confirmed that homogenization is already treated as a recommended function in a later chapter, and that WMO-No. 1131 lists it as recommended rather than required. The team agreed to remove "homogenization" from Figure 1 and from the introductory text in Chapter 1, retaining only "long-term climate data management" in that context. Denis was asked to raise a GitHub issue proposing updated text and any other concerns (e.g., normals data, return periods).

William noted the CDMS definition in WMO-No. 1131. Denis confirmed that WMO-No. 1238 is the authoritative source. David shared the following definition from WMO-No. 1238 in the meeting chat:

____
_Climate data management system (CDMS). A system that facilitates the effective archival, management, analysis, delivery and utilization of a wide range of climate data. According to Climate Data Management System Specifications (WMO-No. 1131), a CDMS combines both IT and governance components. It utilizes a set of interrelated modules that perform specific functions with the data. The purpose of a CDMS is to assist a Member, often through its NMHS, to manage its climate record securely and sustainably._
____

William and Denis agreed this definition should be adopted in the document. As it is too long to use verbatim, William was asked to propose a one-line summary for use in the introductory text.

==== 2.6.3 Discussion: Document Scope

William sought clarification on whether the document describes requirements for a self-contained software package or a broader system. David explained that the CDMS is not a single software package but a system comprising multiple components (database, data catalogue, API, file storage, etc.). This document specifies what the system must do functionally, not how it should be implemented.

=== 2.7 How to Contribute

Ian outlined three levels of GitHub contribution for team members, from simplest to most direct:

. Create a new issue via the provided link, describing the change or concern.
. Find an existing issue for the relevant chapter or section and add a comment.
. Navigate to the relevant text file in the repository, click the edit (pen) icon, make the change, and submit a pull request.

David confirmed that branch protection is in place, so no changes can be accidentally merged without review — team members should not be concerned about making mistakes. A fourth option for those unfamiliar with, or unable to work with due to e.g. institutional policies,  GitHub is to send proposed changes by email to Ian or David, who will add them on their behalf.

David noted that team members who are not on GitHub can still volunteer to review a chapter by email, and their assignment will be tracked in GitHub by Ian or David.
